Data Flow — Filter Request Lifecycle

1
User selects filter
FilterBar.vue updates one of four reactive refs in useFilters.js: selectedPeriod, selectedLocation, selectedCategory, selectedStatus. This composable is a singleton — all views share the same state.
2
View detects change
Each view uses watch([selectedPeriod, selectedLocation, ...], loadData). On change, loadData() is triggered, calling getCurrentFilters() which maps UI state to API params: {warehouse, category, status, month}.
3
Axios API call
api.js sends an HTTP GET to http://localhost:8001/api/<endpoint> with filter params as query strings (e.g. ?warehouse=Tokyo&category=Sensors&month=2025-03).
4
Backend filters in-memory
main.py receives params. apply_filters() iterates the pre-loaded Python list, matching warehouse, category, and status by exact string match. filter_by_month() checks if order_date contains the requested year-month.
5
Pydantic serialization
Filtered dicts are validated and serialized through Pydantic response models (InventoryItem, Order, etc.) before being returned as JSON.
6
Computed properties update
Raw data is stored in refs (allOrders, inventoryItems). Computed properties — statusData, topProducts, categoryData — automatically recalculate and re-render the UI.

Data Entities

Inventory Item
server/data/inventory.json (~40 items)
sku — unique product identifier
name, category, warehouse
quantity_on_hand — current stock level
reorder_point — triggers "Low Stock" status
unit_cost — price per unit
location — shelf/bin within warehouse
Order
server/data/orders.json (~100+ orders)
order_number — e.g. ORD-2025-0001
customer, warehouse, category
items[] — array of {sku, name, quantity, unit_price}
status — Delivered / Shipped / Processing / Backordered
order_date — drives month filter
total_value, actual_delivery
Demand Forecast
server/data/demand_forecasts.json (~10 items)
item_sku, item_name
current_demand — baseline units
forecasted_demand — projected units
trend — increasing / stable / decreasing
period — forecast horizon (e.g. "Next 30 days")
Backlog Item
server/data/backlog_items.json (4 items)
order_id — links to order
item_sku, item_name
quantity_needed vs quantity_available
shortage = quantity_needed − quantity_available
priority — high / medium / low
days_delayed — urgency indicator
Spending
server/data/spending.json + transactions.json
Four cost buckets: procurement, operational, labor, overhead
monthly_spending[] — 12-month series (Jan–Dec 2025)
category_spending[] — Raw Materials, Components, Equipment, Consumables
Each bucket includes YoY _change percentage
Purchase Order
server/data/purchase_orders.json (empty by default)
Created via Dashboard backlog modal
backlog_item_id — links to backlog item
supplier_name, quantity, unit_cost
status — Pending (on creation)
expected_delivery_date, notes
