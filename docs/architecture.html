Data Flow

The application follows a unidirectional data flow pattern with shared reactive state:

1. User interacts with FilterBar component (selects warehouse, category, etc.)
2. FilterBar updates useFilters composable (shared singleton state)
3. View components watch filter changes and trigger API calls
4. getCurrentFilters() converts refs to API query parameters
5. api.js sends HTTP request to FastAPI endpoint
6. Backend applies filters to in-memory data
7. Pydantic validates response data structure
8. Component stores raw data in ref
9. Computed property derives filtered/sorted display data
10. Template renders with v-for using unique keys (item.id)

Four-Filter System

The application features a unified filter system that applies across multiple views:

Time Period Months (2025-01 to 2025-12) or Quarters (Q1-Q4)
Warehouse San Francisco, London, Tokyo
Category Circuit Boards, Sensors, Actuators, Controllers, Power Supplies
Order Status Delivered, Shipped, Processing, Backordered

Filter Implementation:

  • Frontend: Shared reactive state in useFilters.js composable
  • Backend: Query parameter filtering in FastAPI endpoints
  • Special case: Inventory view doesn't support time period filter (no temporal data)
  • Dashboard aggregates all filters for KPI calculations

Data Models

InventoryItem

Stock items tracked across warehouses with reorder points and cost information.

  • id, sku, name, category
  • warehouse, quantity_on_hand, reorder_point
  • unit_cost, location, last_updated

Order

Customer orders with items, status tracking, and delivery dates.

  • id, order_number, customer, items[]
  • status, warehouse, category
  • order_date, expected_delivery, actual_delivery
  • total_value

BacklogItem

Items with insufficient inventory to fulfill orders.

  • id, order_id, item_sku, item_name
  • quantity_needed, quantity_available
  • days_delayed, priority
